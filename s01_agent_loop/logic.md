# s01_agent_loop/code.ts 逻辑分析

## 整体架构

这是整个 AI 编码代理的**最小可运行原型**，共约 170 行，只做一件事：**循环调用 LLM → 执行工具 → 反馈结果**。

## 五个部分

### 1. 初始化（1-48 行）
- 加载 `.env` 环境变量（API Key、Model ID）
- 创建 Anthropic 客户端
- 定义 `SYSTEM` 提示词：告诉模型它是一个编码代理，用 bash 解决问题

### 2. 工具定义（53-61 行）
- 只注册了一个工具：`bash`
- 输入参数：`command`（字符串）
- 后续课程会逐步扩展为 5 个、10 个、20+ 个工具

### 3. 工具执行 `runBash()`（65-84 行）
- **安全过滤**：拦截 `rm -rf /`、`sudo`、`shutdown` 等危险命令
- 通过 `execSync` 在本地 shell 中执行命令
- 超时 120 秒，输出截断到 50000 字符

### 4. 核心循环 `agentLoop()`（88-124 行）

```
┌──────────────┐     ┌──────────┐     ┌────────────┐
│  messages[]  │ ──▶ │   LLM    │ ──▶ │ stop_reason│
│  (对话历史)   │     │ messages │     │ == tool_use?│
└──────┬───────┘     │ .create  │     └──┬────┬────┘
       ▲             └──────────┘        │    │
       │                                是   否
       │            ┌──────────┐         │    │
       │            │ 执行工具  │◀────────┘    │
       │            │ runBash()│              │
       │            └────┬─────┘              │
       │                 │                    │
       └─────────────────┘                    ▼
          tool_result 反馈               输出最终文本
         (循环继续)                      (函数返回)
```

**关键逻辑**：
1. 调用 LLM，传入完整的 `messages` 历史和 `tools` 定义
2. 将 LLM 的响应追加为 `assistant` 消息
3. 如果 `stop_reason !== "tool_use"`，说明模型认为任务完成 → 退出循环
4. 否则遍历 `content` 中的 `tool_use` 块，逐个执行 `runBash()`
5. 将执行结果封装为 `tool_result` 类型，作为 `user` 消息追加回 `messages`
6. **回到步骤 1**，让模型看到工具结果后继续推理

### 5. 入口 `main()`（139-170 行）
- 创建 readline 交互界面
- 死循环等待用户输入（输入 `q` 退出）
- 每次输入作为 `user` 消息加入 `history`
- 调用 `agentLoop(history)` 处理
- 打印模型的最终文本回复

## 设计要点

| 特性 | 实现方式 |
|------|---------|
| 对话记忆 | `messages[]` 数组持续追加，不做裁剪 |
| 工具执行 | 同步 `execSync`，模型调用一个执行一个 |
| 安全边界 | `runBash()` 内的危险命令关键词过滤 |
| 循环终止 | 依赖 `stop_reason`，模型自己决定何时结束 |
| 错误处理 | try/catch 捕获超时和执行错误，不中断循环 |
